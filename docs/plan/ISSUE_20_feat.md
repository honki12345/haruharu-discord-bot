# ISSUE #20 구현 계획

## 목표

- `#체크인체크아웃` 채널의 노이즈를 줄이기 위해 daily message + thread 기반 하루 1회 출석 구조로 전환한다.
- 출석은 slash command가 아니라 오늘의 thread 참여로 인식한다.
- 출석 결과는 봇 이모지 반응으로 빠르게 확인할 수 있게 한다.

## 현재 결정된 정책

- 출석은 하루 1회만 인정한다.
- 기준 시간은 사용자 등록 `waketime`이다.
- 오늘 thread에 남긴 첫 댓글 시간만 판정에 사용한다.
- `waketime -10분 ~ +10분`: 출석
- `waketime +11분 ~ +30분`: 지각
- `waketime +30분 초과`: 결석
- `waketime -10분` 이전: 아직 출석 가능 시간이 아님
- 이미지는 필수가 아니다.

## 확정된 구현 결정

- 출석 저장 구조는 기존 `TimeLog`를 재사용하지 않고 새 모델 `AttendanceLog`로 분리한다.
- `AttendanceLog`는 thread 기반 하루 1회 출석만 담당하고, 기존 `TimeLog`는 레거시 `check-in/check-out` 전용으로 남긴다.
- `too-early` 상태는 공식 출석 기록으로 저장하지 않는다.
- `too-early` 댓글에는 `⏰` 반응만 달고, 출석 가능 시간 이후에 다시 남긴 첫 유효 댓글을 공식 기록으로 사용한다.
- 구현 순서는 `데모 검증 -> AttendanceLog 모델 구현 -> 운영 자동화 -> 집계 전환`으로 고정한다.

## 이모지 정책

- 출석: `✅`
- 지각: `🟡`
- 결석: `❌`
- 아직 이른 시간: `⏰`
- 미등록 사용자 댓글: `❓`

결석 이모지는 두 경우로 나눈다.

- 사용자가 `+30분` 이후에 thread에 댓글을 남긴 경우: 해당 메시지에 `❌` 반응 가능
- 사용자가 아예 댓글을 남기지 않은 경우: 반응할 메시지가 없으므로 일일 출석 리포트나 요약 메시지에서 `❌`로 표시

## 단계별 계획

## 구현 원칙

- 데모에서 검증된 흐름만 운영 채널로 확장한다.
- 한 번에 전체 전환하지 않고, `메시지 인식 -> 저장 구조 -> 집계 -> 기존 명령 축소` 순서로 나눈다.
- 운영 정책이 확정되기 전까지는 기존 `/check-in`, `/check-out` 흐름을 즉시 제거하지 않는다.
- 댓글 기반 출석은 "첫 댓글 시각"이 진실 원본(source of truth)이 되도록 설계한다.
- 결석은 반응 이벤트가 아니라 집계 이벤트에서 최종 확정한다.

## 아키텍처 변경 포인트

### 현재 구조

- 출석 입력: slash command (`/check-in`, `/check-out`)
- 출석 저장: `TimeLog` 1일 2건 전제
- 출석 집계: `ready.ts`의 13:00 스케줄러

### 목표 구조

- 출석 입력: daily message에 연결된 thread의 첫 댓글
- 출석 저장: 1일 1건 기준 출석 기록
- 출석 판정: `messageCreate` 시점의 1차 판정 + 일일 집계 시점의 결석 확정
- 출석 피드백: 댓글 메시지 이모지 반응

### 파일 단위 예상 변경

- `src/index.ts`
  - 메시지 intent 유지 및 운영 이벤트 등록 확인
- `src/events/messageCreate.ts`
  - 현재 demo 전용 분기에서 운영 분기까지 확장
- `src/events/ready.ts`
  - daily message 생성 스케줄 추가
  - 일일 집계 로직을 하루 1회 출석 기준으로 전환
- `src/attendance.ts`
  - 출석 판정, 이모지, 상태 문자열의 단일 진입점 유지
- `src/repository/*`
  - 출석 전용 저장 구조 추가 또는 `TimeLog` 전환
- `docs/PROJECT.md`, `docs/USER_STORIES.md`
  - 정책, 흐름, 데이터 모델 문서 동기화

범위:
- 관리자용 `/demo-daily-message` 커맨드 추가
- 테스트 채널에 daily message와 demo thread 생성
- `messageCreate` 이벤트로 thread 댓글 감지
- 등록 사용자 첫 댓글 기준으로 이모지 반응
- 단위 테스트 추가

완료 기준:
- 테스트 채널에서 demo thread가 생성된다
- 등록된 사용자가 댓글을 남기면 시각에 맞는 이모지가 반응한다
- 같은 사용자의 두 번째 댓글은 출석 판정에 사용하지 않는다
- 운영 코드에 영향을 주지 않고 격리된 상태로 검증 가능하다

산출물:
- `/demo-daily-message` 커맨드
- demo 전용 `messageCreate` 처리
- 출석 판정 유틸리티 테스트

검증 방식:
- 단위 테스트
- 테스트 채널 수동 검증
- 로그 확인

### Phase 1. 출석 데이터 구조 전환

목적:
기존 `check-in/check-out` 2건 구조를 하루 1건 출석 구조에 맞게 정리한다.

작업:
- 현재 `TimeLog` 구조를 유지할지 새 출석 전용 모델을 만들지 결정
- 하루 1건 기준 저장 컬럼 정의
- 상태값(`attended`, `late`, `absent`) 저장 전략 정리
- 중복 댓글 처리와 첫 댓글 고정 로직 반영

검토 포인트:
- 기존 데이터와의 호환
- 기존 리포트 로직 수정 범위
- 테스트 마이그레이션 비용

선택지:

1. `TimeLog` 재사용
- 장점: 기존 테이블 재사용
- 단점: 현재 의미가 `check-in/check-out` 2건 구조라 의미 충돌이 큼

2. 새 출석 전용 모델 추가
- 예시: `AttendanceLog`
- 장점: 정책이 명확하고 마이그레이션/집계 로직이 단순해짐
- 단점: 초기 구현량 증가

권장:
- 새 출석 전용 모델 `AttendanceLog`를 채택한다.
- 권장 컬럼 예시:
  - `userid`
  - `username`
  - `yearmonthday`
  - `threadid`
  - `messageid`
  - `commentedat`
  - `status`
  - `createdAt`
  - `updatedAt`

저장 정책:
- `attended`, `late`, `absent`만 공식 저장 대상
- `too-early`는 반응만 하고 저장하지 않음
- 동일 사용자의 당일 첫 공식 기록만 유지

완료 기준:
- 운영 저장 구조가 확정된다
- 첫 댓글 중복 방지 방식이 확정된다
- 집계 로직이 어떤 테이블을 기준으로 돌지 확정된다

### Phase 2. 운영 daily message 자동화

작업:
- `ready.ts` 또는 별도 스케줄러에서 매일 daily message 생성
- 오늘 thread 식별자 저장 또는 조회 전략 추가
- auto-archive 정책 결정
- 운영 채널과 테스트 채널 분리

상세 작업:
- daily message 생성 시간을 상수화
- 당일 thread가 이미 있으면 재생성하지 않도록 idempotent 처리
- `YYYYMMDD -> messageId/threadId`를 저장할지, thread 이름 조회로 충분할지 결정
- 봇 재시작 후에도 오늘 thread를 다시 찾을 수 있어야 함

권장:
- 초기에는 `thread 이름 + parent channel + 오늘 날짜` 조회로 시작
- 이후 불안정하면 메타 테이블 또는 JSON 저장으로 전환

완료 기준:
- 봇 재시작 후에도 오늘 thread를 식별할 수 있다
- 중복 daily message 생성이 방지된다
- 테스트 채널과 운영 채널이 명확히 분리된다

### Phase 3. 출석 집계 및 리포트 전환

작업:
- 기존 13:00 출석 집계 로직을 하루 1회 출석 기준으로 수정
- 결석자 판정 시 미댓글 사용자도 포함
- 요약 메시지에 상태별 이모지 반영
- 월말/누적 통계 영향 점검

상세 작업:
- 출석자: `attended`
- 지각자: `late`
- 결석자:
  - 댓글이 없던 사용자
  - `absent` 상태로 기록된 늦은 댓글 사용자
- late/absence 카운트 업데이트 기준 재정의
- 휴가(`vacances`) 처리와의 관계 재검증

완료 기준:
- 기존 13:00 리포트가 새 정책 기준으로 정확히 출력된다
- 미댓글 사용자도 결석으로 반영된다
- 월말 생존 명단 로직이 새 결석 카운트와 일관된다

### Phase 4. 기존 slash command 축소

작업:
- `/check-in`, `/check-out`를 비활성화할지, 안내 메시지로 전환할지 결정
- 전환 기간 동안 backward compatibility 정책 정리
- 관련 문서와 테스트 정리

권장 전환:

1. 1차 배포
- 기존 명령 유지
- 사용 시 thread로 안내

2. 2차 배포
- 명령은 남기되 출석 기록은 하지 않고 안내만 수행

3. 최종
- 명령 제거 또는 deprecated 처리

완료 기준:
- 운영 공지가 준비된다
- 구버전 사용자가 새 흐름으로 이동할 수 있다
- 문서/테스트가 새 정책 기준으로 정리된다

## 이벤트 흐름 설계

### A. daily message 생성 흐름

1. 스케줄러 트리거
2. 오늘 운영 채널 확인
3. 오늘 날짜의 기존 thread/message 존재 여부 확인
4. 없으면 daily message 전송
5. thread 생성
6. 안내 메시지 전송
7. 생성 로그 남김

### B. 출석 댓글 인식 흐름

1. `messageCreate` 수신
2. 봇 메시지/DM/일반 채널 메시지 제외
3. 오늘 출석 thread인지 확인
4. 등록 사용자 조회
5. 당일 첫 댓글인지 확인
6. 댓글 시각으로 상태 판정
7. `too-early`면 `⏰` 반응만 추가하고 종료
8. 공식 출석 상태(`attended`, `late`, `absent`)면 `AttendanceLog` 저장 또는 유지
9. 이모지 반응 추가
10. 로그 기록

### C. 일일 집계 흐름

1. 13:00 스케줄러 트리거
2. 오늘 등록 사용자 전체 조회
3. 당일 출석 기록 조회
4. 미댓글 사용자 결석 확정
5. late/absence 카운트 반영
6. 결과 메시지 전송
7. 월말 생존 명단 로직과 연계

## 데모 시나리오

1. 관리자 계정으로 `/demo-daily-message` 실행
2. 테스트 채널에 daily message와 `YYYY-MM-DD 출석-demo` thread 생성 확인
3. 등록된 사용자가 thread에 첫 댓글 작성
4. 봇이 댓글 시간 기준으로 `✅`, `🟡`, `❌`, `⏰` 중 하나를 반응
5. 같은 사용자가 추가 댓글 작성 시 새 판정이 붙지 않는지 확인
6. 미등록 사용자가 댓글 작성하면 `❓` 반응 확인

- 단위 테스트:
  - 출석 시간 판정 함수
  - demo command thread 생성
  - messageCreate 첫 댓글 판정 및 이모지 반응
  - 날짜 경계 조건
  - 중복 댓글 방지
  - 미등록 사용자 처리
- 통합 테스트:

  - 실제 thread 댓글 반응 확인
  - 운영 channel/thread 생성 리허설
  - 봇 재시작 후 thread 재탐색 확인

## 운영 전환 체크리스트

- 운영 채널 ID와 테스트 채널 ID가 분리되어 있는가
- 메시지 intent가 운영 환경에도 반영되어 있는가
- daily message 중복 생성 방지 장치가 있는가
- 봇 재시작 시 오늘 thread를 다시 찾을 수 있는가
- 13:00 집계가 새 저장 구조를 읽는가
- 휴가/공휴일 규칙과 충돌하지 않는가
- 기존 `/check-in`, `/check-out` 사용자가 혼란 없이 이동할 수 있는가

## 롤백 전략

- 운영 문제 발생 시 daily message 자동 생성만 끄고 기존 slash command 흐름으로 임시 복귀한다.
- `messageCreate` 운영 분기를 비활성화하고 demo 전용 흐름만 남긴다.
- 새 출석 저장 모델을 도입했다면 읽기 전용으로 유지하고 집계는 기존 방식으로 되돌린다.

## 리스크

- 메시지 기반 출석은 Discord intent 설정에 의존한다.
- 결석자는 메시지가 없으므로 "메시지 반응"만으로는 완결되지 않는다.
- 기존 `TimeLog` 구조가 두 단계 출석을 전제로 하고 있어 저장 구조 변경이 필요할 수 있다.
- 운영 채널에서 전환 시 기존 사용자 습관 때문에 초기 안내가 중요하다.
- 날짜 경계 처리와 봇 재시작 상황에서 오늘 thread 식별이 어긋날 수 있다.
- Discord API 제한 또는 thread 조회 실패 시 자동화가 중복 실행될 수 있다.
